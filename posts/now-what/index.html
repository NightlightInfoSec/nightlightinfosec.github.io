I Just Bought A Domain: Now What

A Cautionary Tale

Writing a blog post is such a great way to solidify your thoughts.
When you write, you really have to think about what you are saying.

When you buy a domain name though, sometimes you are living in the moment and forget about being logical.

Planning

After I bought my domain name, I wanted to jump right in and start learning the ins and outs of creating a website.
I was so caught up in the moment that I overlooked one of the most important steps: Planning. If you are reading this, congratulations you just bought a domain name or are thinking of buying one. Hindsight is 20⁄20, and if I could go back and start over I probably would. But I don’t own a time machine, so we can all learn from my mistakes.

Choosing the right format - What are you trying to achieve and what features do you want (be specific) ?

I chose to go with a blog because I believe that writing will help me think through and understand hard problems. To be honest, I also wanted a way to showcase my skills to any future employers too.

I wish I would have spent more time thinking about some of the smaller details of what future me would have wanted.
Like:

  • How are people going to comment on posts to keep me in check if I say something wrong?
  • How can someone search my blog for specific topics?
  • Is there a way to archive posts?
  • What other integrations are available?

I can still figure some of this out later and implement changes, but if I would have already thought about this in the beginning I could have maybe saved myself some headaches.

Hosting

You have a lot to think about when it comes to hosting
Like:

  • How much money do I want to spend?
  • Is uptime important to my site?
  • What kind of support do I need?
  • Does the hosting provider offer site templates?

I didn’t want to spend any money on hosting, and thus forgot about anything else that a hosting provider might bring to the table. I instead decided to go with GitHub Pages, because it’s free and I have used GitHub for other projects in the past.

Hosting with GitHub Pages

First you will need to set up a GitHub account by following these Instructions.

Next create a new project/repository




Name the repository UserName.github.io





This is where you would start using git to upload content to your new repository. But maybe you’re like me and are brand new to using version control. Fortunately, GitHub offers a git client called GitHub Desktop that works great





Sign into the Desktop Client





Go back to your GitHub repository and click “Set up in Desktop”(I created a new project for images since my page already exists)




Click “Open GitHubDesktop”





Select the local path you would like to host your files locally and click “Clone”





Go back to your GitHub repository and Select “Settings”





Scroll down to “GitHub Pages” and select Source: Master Branch



Here you also have the option to select a theme. These themes are rather limiting though, so I opted for a theme with a little more substance.

Design and Implementation

GitHub Pages hosts static web pages, so if you want something a little more fancier, you will need to choose a different hosting provider.

There are quite a few platforms for generating static web pages.

The most popular platform is probably Jekyll. However, it requires installing the Ruby programming language which I don’t have that much interest in learning (Choose what interests you). To be fair, they do have a lot of resources and themes that are useful to newcomers.

Instead I chose Hugo because it’s written in a language I’m intersted in learning, GO! They also have some pretty good resources and themes.

I wont go through installing Go or Hugo, but I will go over some of the things I thought were not documented well.

PS: Please do not confuse the face that I didn’t want to install Ruby over GO becasue you have to learn those languages. That is just the language they are written in.

Tip Number 1

  • Unless specified in your theme’s templates, all Posts are in draft form. Change this to false if you’re planning to post immediately.

    ---
title: "First Post"
date: 2019-10-31T14:28:26-05:00`
draft: true
---

Tip Number 2

  • You only need to upload the contents of the /Public folder to GitHub

    /Public  is created once you run the Hugo command

Tip Number 3

  • Most themes come with an exampleSite folder. Use the config.toml file as a guide to configure your own.

Push your files up to your GitHub Master repository and head over to UserName.github.io

Your page should be live now on GitHub!!

Domain Name System (DNS)

DNS translates your new domain name into an IP address that your web browser understands. Your hosting provider might be able to host your DNS for you for a fee.

For my DNS hosting, I went with Cloudflare because they offer a free plan.

I changed the CNAME to Name=NewDomainName Content=UserName.github.io

Back over on GitHub, in “settings” –> “GitHub Pages” I added my domain name to the “Custom Domain” field and clicked “Save”

This adds a new file to your GitHub repository called CNAME

The key now patience because it can take a while for this information to propagate.

Hopefully by now everything is working and you are now able to enter your domain name into the address bar and be redirected to your GitHub Page!

I’m still learning, but I hope you found some value in this post!
